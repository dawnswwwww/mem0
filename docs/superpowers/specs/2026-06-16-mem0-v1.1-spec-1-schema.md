# mem0 v1.1 spec-1: Schema layer

**Date:** 2026-06-16
**Status:** Approved (pending spec review)
**Scope:** v1.1 spec-1 of 4 — schema migration framework + tags FTS denormalization + CJK tokenization. Other v1.1 features (soft delete / config / concurrency) live in their own specs.

## 1. Goal

Lay the foundation for all future schema changes by introducing (a) a migration framework that tracks schema version via `PRAGMA user_version`, and ship two v1 → v2 changes in the same migration: (b) denormalize tags into a plain-text `tags_text` column so FTS5 no longer indexes raw JSON, and (c) switch the FTS5 tokenizer to `trigram` so CJK content is searchable. v1.1 spec-1 has no new CLI surface and no new user-facing behavior beyond "Chinese now works" and "tag search is more robust."

## 2. Non-Goals (spec-1)

- Soft delete (`archived` lifecycle, `mem0 archive` command) → spec-2
- `mem0 config` subcommand → spec-3
- Multi-process safety / lockfile → spec-4
- Replacing the FTS5 `content` column behavior
- Changing the `tags` JSON column (it remains the source of truth; `tags_text` is a denormalized cache)
- Backfilling or cleaning up DBs from versions older than v1 (we ship from v1)

## 3. Architecture

```
store::db::migrate(conn)
  ↓
BEGIN TRANSACTION
  version = PRAGMA user_version
  if version < 1: apply_v1_initial(conn)   ← unchanged from v0.1.0
  if version < 2: apply_v2_v1_1(conn)      ← NEW
COMMIT
```

`apply_v1_initial` is a refactor of the existing v0.1.0 inline DDL into a callable function so the migration runner can branch on `PRAGMA user_version`. The SQL itself is unchanged (still `CREATE TABLE IF NOT EXISTS ...` etc.). `apply_v2_v1_1` runs entirely inside the same transaction as the version gate, so a partial failure leaves the DB at v1 with no half-applied schema.

## 4. Schema v2 (diff vs v1)

### 4.1 `memories` table

```sql
-- v1 had: tags TEXT NOT NULL DEFAULT '[]'  (JSON array)
-- v2 adds:
ALTER TABLE memories ADD COLUMN tags_text TEXT NOT NULL DEFAULT '';

-- Why both columns?
--   tags       = source of truth (JSON, preserves order, escaping)
--   tags_text  = denormalized cache for FTS5 (plain text, predictable tokenization)
```

`tags_text` is derived from `tags` at insert/update time by `store::memories::insert()` and the migration's backfill. Format: **space-separated, lowercased**. Example: `tags = ["Preference","whiskey"]` → `tags_text = "preference whiskey"`. The space separator works for trigram tokenization (trigrams cross whitespace) and is human-debuggable.

### 4.2 FTS5 virtual table

```sql
-- v1: USING fts5(content, tags, content='memories', content_rowid='rowid')
-- v2: USING fts5(content, tags_text, content='memories', content_rowid='rowid', tokenize='trigram')
```

The `tags` column is dropped (it indexed raw JSON). `tags_text` is the new FTS column. `tokenize='trigram'` enables substring search on CJK content.

### 4.3 Triggers (recreated to match new schema)

```sql
CREATE TRIGGER memories_ai AFTER INSERT ON memories BEGIN
  INSERT INTO memories_fts(rowid, content, tags_text) VALUES (new.rowid, new.content, new.tags_text);
END;
CREATE TRIGGER memories_ad AFTER DELETE ON memories BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, content, tags_text) VALUES('delete', old.rowid, old.content, old.tags_text);
END;
CREATE TRIGGER memories_au AFTER UPDATE ON memories BEGIN
  INSERT INTO memories_fts(memories_fts, rowid, content, tags_text) VALUES('delete', old.rowid, old.content, old.tags_text);
  INSERT INTO memories_fts(rowid, content, tags_text) VALUES (new.rowid, new.content, new.tags_text);
END;
```

Trigger bodies reference `tags_text` instead of `tags`. `store::memories::insert` populates `tags_text` so the trigger sees a non-empty value.

## 5. Migration Sequence (`apply_v2_v1_1`)

| Step | SQL | Idempotency |
|---|---|---|
| 1 | `ALTER TABLE memories ADD COLUMN tags_text TEXT NOT NULL DEFAULT ''` | Guarded by `PRAGMA table_info` column check |
| 2 | `UPDATE memories SET tags_text = <derived from tags>` (every row) | Idempotent (overwrites with same value) |
| 3 | `DROP TABLE IF EXISTS memories_fts` | `IF EXISTS` |
| 4 | `CREATE VIRTUAL TABLE memories_fts USING fts5(content, tags_text, content='memories', content_rowid='rowid', tokenize='trigram')` | Fresh table; no need for `IF NOT EXISTS` after DROP |
| 5 | `INSERT INTO memories_fts(rowid, content, tags_text) SELECT rowid, content, tags_text FROM memories` | Rebuilds from base table |
| 6 | `DROP TABLE IF EXISTS memories_ai` / `_ad` / `_au` | Old triggers reference `tags` column |
| 7 | `CREATE TRIGGER ...` (new versions) | New triggers reference `tags_text` |
| 8 | `PRAGMA user_version = 2` | Sets the gate |

All 8 steps are inside one `BEGIN ... COMMIT`. Any failure triggers `ROLLBACK`, returning the DB to its v1 state.

### 5.1 Tags-text derivation

```sql
UPDATE memories
SET tags_text = lower(
  replace(replace(replace(replace(replace(tags, '[', ''), ']', ''), '"', ''), ',', ' '), '  ', ' ')
);
-- Walks: JSON array → space-separated → trim double spaces → lowercase
-- Example: '["A","B C"]' → 'a b c'
```

The `replace('  ', ' ')` chain handles consecutive separators from empty strings (e.g. `'["a",""]'` → `'a '` not `'a  '`). Two passes are sufficient for any depth-1 JSON array.

## 6. Module-level Changes

| File | Change |
|---|---|
| `src/store/migrations.rs` | Add `pub fn apply_v1_initial(conn: &Connection) -> MemResult<()>` (extracts the v0.1.0 DDL from `migrate()` into a callable) and `pub fn apply_v2_v1_1(conn: &Connection) -> MemResult<()>`. The `V1_SCHEMA` const is removed (its content moves into `apply_v1_initial`). |
| `src/store/db.rs` | `migrate()` reads `PRAGMA user_version`, dispatches to `apply_v1_initial` / `apply_v2_v1_1` based on current version. The existing `BEGIN/COMMIT/ROLLBACK` wrapper stays. |
| `src/store/memories.rs` | `insert()` computes `tags_text` from `tags` inline (no `MemoryDraft` field change) and writes both columns in one `INSERT`. `row_to_item` reads `tags_text` for completeness but doesn't expose it via `MemoryItem` (it's an internal cache, not user-facing). |
| `src/output/format.rs` | No change. `tags_text` is not serialized to JSON. |
| `src/cli/*` | No change. |
| `Cargo.toml` | No new deps. |

## 8. Error Handling

| Failure | Behavior |
|---|---|
| `apply_v2_v1_1` step 1 fails (e.g. column-add disallowed) | Transaction rolls back; DB stays at v1; next `migrate()` retries. |
| Step 2 fails (UPDATE syntax error) | Same — rollback. |
| Step 4 fails (CREATE VIRTUAL TABLE with bad tokenizer args) | Same — but this should not happen since we control the SQL. |
| Step 5 fails (out-of-disk during reindex) | Same — partial reindex is rolled back. `memories_fts` doesn't exist after rollback, so until user re-runs migrate, search will fail with "no such table" (surfaced as `MemError::Storage`). This is the right behavior — better to fail loudly than to silently search against a stale or missing index. |
| App calls `search` before migrate completes | Not possible — `migrate()` is called from `cli::run` before any subcommand executes. |

## 9. Open Questions (deferred, not blocking)

- **Multi-process migration.** If two `mem0` processes run `migrate()` simultaneously, both could attempt `PRAGMA user_version = 2`. SQLite serializes write transactions, so one will succeed and the other will see v2 already (skip step). Safe in practice; explicit lockfile is spec-4's job.
- **Migration logging.** Should `migrate()` print to stderr when it applies v2? v0.1.0 is silent. v1.1 stays silent to avoid surprising users with non-JSON stderr. A future `--verbose` flag can opt in.
- **FTS5 shadow tables after migration.** When we DROP `memories_fts`, the FTS5 shadow tables (`memories_fts_config`, etc.) go with it. The new CREATE recreates them. No leakage.

## 10. Definition of Done (spec-1)

- All 8 migration steps execute in a single transaction with no observable side effects on failure.
- `PRAGMA user_version` reads `2` after migration on a v1 DB.
- All 78 existing v0.1.0 tests still pass.
- 7 new tests pass (5 migration + 2 store_memories).
- `cargo clippy --all-targets -- -D warnings` clean.
- `mem0 search "威士忌"` on a Chinese-tagged DB returns the matching row (verified manually; not in CI test due to fixture-data overhead).
